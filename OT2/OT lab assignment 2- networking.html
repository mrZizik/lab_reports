Installing DNS server on host machine

We chose bind9 as our dns server. Installation is quite easy.



apt install bind9
service bind9 stop

We’ll just put here configuration files of server and zone.

/etc/bind/named.conf.options















dnssec-validation auto;
auth-nxdomain no;
listen-on-v6 { any; };
forwarders {
    8.8.8.8;
    8.8.4.4;
    };

forward only;

listen-on {
    192.168.56.1;
    188.130.155.42;
    };

/etc/bind/named.conf.local





zone "vm.st9" {
	type master;
	file "/etc/bind/db.vm.st9";
}

How can you prevent a MITM attack on a network and protocol level?

If you can not control the network you will not be able to do a lot. That is why you should never use public networks for operating with important data. It is good to encrypt as much data as you can.

There’re many methods of MITM. If we speak about ARP, we have several ways to prevent sniffing our traffic besides encryption.

  1. We can divide our network to segments, because ARP attack can not be processed on hosts in different L2 domains.
  2. We can set static ARP tables for our hosts, so attacker wouldn’t be able to rewrite our table.

Also IDSs like Snort can notify you when MITM attacks are taking place. The bad side is that you may receive false alerts, because it is hard to write accurate rules.

There are tools which use the advanced address resolution protocol (like XARP or ARPOn) and measures like implementing dynamic host configuration protocol (DHCP) snooping on switches can limit or prevent ARP spoofing.

Covert Channels

Many tools for dns/icmp tunneling are available. We used iodine and icmpttunnel.

DNS tunnel

Installation:






wget http://code.kro.se/iodine/iodine-0.7.0.tar.gz
unzip iodine-0.7.0.tar.gz
cd iodine-0.7.0.tar.gz
make
sudo make install

On server:

./iodined 10.99.99.1/24 vm.st9

On client:
Firstly, we add ip route for server host, because we need our iodine client иу фиду to connect to iodined server.

ip route add 192.168.56.1/32 dev eth0

Then we connect our tunnel:

./iodine 192.168.56.1 vm.st9

After that we add default route for our new tunnel

ip route add default via 10.99.99.1

Then we need to configure masquerade at server’s ip tables:

iptables -t nat -A POSTROUTING -o dns0 -j MASQUERADE

For testing purposes we ran mtr on client and can see how our traffic is going:

mtr 192.168.56.103

Detection

Our dns tunnel wasn’t detected by snort by default, bu we wrote our rules based on payload size:

alert udp any any -> any 53 (dsize: >90; msg: "DNS tunnel"; sid: 1000005; rev: 1)

ICMP tunnel




git clone https://github.com/DhavalKapil/icmptunnel
cd icmptunnel
make

On server:

./icmptunnel -s 10.0.1.1

On client:

Firstly we edit config file:

client.sh





...
ip route add 192.168.56.1/32 dev eth0
route add default gw 10.0.1.1 tun0
...

Then we start our tunnel.

./icmptunnel -c 192.168.56.1 &

We can see how it is working in that title. We go to 192.168.56.103 through 10.0.1.1

Detection

ICMP tunnel is also not detected by snort by default. There is rule for detection such method.

alert icmp any any -> any any (dsize: >100; msg: "ICMP tunnel"; sid: 10000004; rev: 1)

Prevention

Usage of covert channels can’t be prevented by 100%, but it can be reduced via filtering some unusual traffic going through some channels. For example by packet size, like we’ve done in snort.
